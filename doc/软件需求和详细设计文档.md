# 软件需求文档（SRS）

## 1. 项目概述

### 1.1 项目背景

随着电商平台的蓬勃发展，用户在拼多多等平台上的收藏商品日益增多，原生平台的收藏功能缺乏有效的分类管理能力，导致用户难以系统性地规划采购项目。【心愿】应用旨在解决这一痛点，通过一键同步电商平台收藏商品，帮助用户实现清单式采购管理。

### 1.2 项目目标

- 提供拼多多收藏商品的一键同步功能
- 支持用户创建采购项目（心愿），自主分类管理商品
- 实现采购进度追踪，标记项目完成
- 提供清晰、美观、流畅的用户体验

### 1.3 目标用户

- 经常在拼多多购物的用户
- 有计划性采购需求的消费者（如装修采购、节日备货、批量采购等）
- 希望系统化管理购物清单的用户

### 1.4 项目范围

| 项目阶段     | 范围                                                         |
| ------------ | ------------------------------------------------------------ |
| V1.0（当前） | 拼多多账号授权登录、收藏商品同步、项目创建与管理、商品分类、采购清单查看 |
| V1.x（后续） | 更多电商平台接入（京东、淘宝等）、数据导出、分享功能         |
| V2.0（远期） | 智能推荐、价格监控、预算管理                                 |

## 3. 用户场景

### 3.1 核心场景

**场景一：用户首次使用**
1. 用户下载并打开应用
2. 应用展示登录页面，要求使用拼多多账号授权登录
3. 用户完成拼多多OAuth2.0授权，获取access_token
4. 应用自动同步该用户的拼多多收藏商品列表
5. 进入首页

**场景二：浏览和创建项目**
1. 用户进入首页，浏览已有的“心愿”项目列表
2. 若无项目或需新建，点击“创建心愿”按钮
3. 设置项目名称、预置场景分类、标记信息
4. 浏览从拼多多同步过来的收藏商品列表
5. 商品列表展示：缩略图、名称、价格、所属项目标签
6. 通过搜索、筛选、智能推荐等方式查找商品
7. 勾选商品加入当前项目
8. 在项目中创建子分类，将商品移动到对应分类下
9. 保存项目

**场景三：管理项目清单**
1. 用户点击已创建的项目
2. 查看项目详情：商品列表、采购量统计、单价合计
3. 可进行编辑、删除、置顶等操作
4. 所有商品采购完成后，标记项目为“已完成”

**场景四：查看商品详情**
1. 用户点击项目中的任意商品
2. 应用展示商品基础信息（名称、缩略图、细节图、价格等）
3. 点击后跳转至拼多多App查看商品完整详情

## 3. 软件架构设计

### 3.1 架构概览

采用**Clean Architecture + 分层架构**设计，遵循**关注点分离**原则。应用分为三层：

```
┌─────────────────────────────────────────────────────┐
│                   Presentation Layer                │
│  (Screens / Widgets / ViewModels / State Management) │
├─────────────────────────────────────────────────────┤
│                    Domain Layer                     │
│  (Entities / Use Cases / Repository Interfaces)      │
├─────────────────────────────────────────────────────┤
│                     Data Layer                      │
│  (Repositories Impl / Data Sources / Local DB / API) │
└─────────────────────────────────────────────────────┘
```

### 3.2 分层详解

#### 3.2.1 Presentation Layer（展示层）

- **职责**：UI渲染、用户交互、状态展示
- **组成**：
  - Screens：页面级别的Widget
  - Widgets：可复用的UI组件
  - ViewModels（通过Riverpod Providers）：状态管理和业务逻辑调用
- **原则**：Widget应保持“瘦”状态，尽可能无逻辑

#### 3.2.2 Domain Layer（领域层）

- **职责**：核心业务逻辑、业务规则
- **组成**：
  - Entities：纯数据类（User、Project、Product、Category等）
  - Use Cases：业务用例（GetProjects、CreateProject、SyncProducts等）
  - Repository Interfaces：抽象接口定义
- **原则**：纯Dart代码，不依赖Flutter框架和外部库

#### 3.2.3 Data Layer（数据层）

- **职责**：数据源管理、数据持久化
- **组成**：
  - Repository Implementations：仓储接口的具体实现
  - Data Sources：远程API数据源、本地数据库数据源
  - Models：数据传输对象（DTO），与Entity转换
- **原则**：单一数据源（Single Source of Truth）

### 3.4 架构原则

1. **依赖方向**：Presentation → Domain ← Data（依赖倒置）
2. **单一职责**：每个类/模块只负责一个功能
3. **开闭原则**：对扩展开放，对修改封闭
4. **接口隔离**：使用细粒度接口，避免胖接口
5. **单一数据源**：每个数据类型只有一个可信来源

## 4. 模块详细设计

### 4.1 认证模块（Auth）

#### 4.1.1 功能描述
处理拼多多OAuth2.0授权登录、Token管理和登出。

#### 4.1.2 核心类设计

**AuthRepository（接口）**
```dart
abstract class AuthRepository {
  Future<Result<User>> login();
  Future<Result<void>> logout();
  Future<Result<String>> getAccessToken();
  Future<Result<String>> refreshToken();
  Future<bool> isLoggedIn();
}
```

**AuthRepositoryImpl（实现）**
- 依赖：PddApiDataSource、SecureStorage
- 职责：实现OAuth2.0授权流程，管理Token存储与刷新

#### 4.1.3 授权流程

```
1. 用户点击“拼多多授权登录”
2. 打开拼多多OAuth2.0授权页面
```

## 8. 状态管理设计

### 8.1 状态管理方案：Riverpod 3.0

选择Riverpod作为状态管理方案，理由如下：

1. **无需BuildContext**：状态与UI完全解耦，便于测试
2. **编译安全**：通过@riverpod代码生成，消除运行时错误
3. **强大的测试能力**：支持overrideWithValue()模拟状态
4. **异步支持**：AsyncValue自动处理loading/error/data状态

### 8.2 Provider分层设计

```dart
// 1. 全局Provider（core/di/providers.dart）
final dioProvider = Provider<Dio>((ref) => Dio());
final secureStorageProvider = Provider<SecureStorage>((ref) => SecureStorage());

// 2. Repository Provider
final authRepositoryProvider = Provider<AuthRepository>((ref) {
  return AuthRepositoryImpl(
    apiDataSource: ref.watch(pddApiDataSourceProvider),
    secureStorage: ref.watch(secureStorageProvider),
  );
});

// 3. UseCase Provider
final loginUseCaseProvider = Provider<LoginUseCase>((ref) {
  return LoginUseCase(ref.watch(authRepositoryProvider));
});

// 4. ViewModel Provider（StateNotifierProvider）
final projectListProvider = StateNotifierProvider<ProjectListNotifier, AsyncValue<List<Project>>>((ref) {
  return ProjectListNotifier(ref.watch(getProjectsUseCaseProvider));
});
```

### 8.3 状态分类

| 状态类型     | 管理方式                  | 示例                   |
| ------------ | ------------------------- | ---------------------- |
| 应用全局状态 | Provider                  | 用户登录状态、主题模式 |
| 页面状态     | StateNotifierProvider     | 项目列表、商品列表     |
| 局部UI状态   | StateProvider             | 弹窗显隐、Tab切换      |
| 异步状态     | FutureProvider/AsyncValue | API请求结果            |

```dart
sealed class Failure {

class NetworkFailure extends Failure {
```

### 9.2 错误处理策略

| 错误类型     | 处理方式               | 用户反馈                |
| ------------ | ---------------------- | ----------------------- |
| 网络错误     | 重试机制（最多3次）    | Toast提示“网络连接异常” |
| 认证错误     | 自动刷新Token          | 若刷新失败，跳转登录页  |
| 服务器错误   | 记录日志，展示友好信息 | Toast提示“服务暂不可用” |
| 本地存储错误 | 降级方案               | Toast提示“数据加载失败” |

## 11. 安全设计

### 11.1 Token安全

- Access Token使用 `flutter_secure_storage` 加密存储（Android Keystore / iOS Keychain）
- Token刷新机制：提前5分钟刷新
- 登出时清除所有本地敏感数据

- 敏感信息（Client Secret）不写入代码，使用环境变量或配置服务
